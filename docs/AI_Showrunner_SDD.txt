1. Executive Summary
AI Showrunner is an autonomous short-drama production agent submitted to the Qwen Cloud Global AI Hackathon under Track 2 ("AI Showrunner"). Given a single free-text prompt from a user, the system produces a complete short drama — script, storyboard, generated video shots, and an edited final cut — with no further human input. All LLM reasoning routes through Qwen Cloud (DashScope) models; all video generation uses the HappyHorse family (t2v / i2v) via the DashScope video-synthesis endpoint.
The system is architected around a pattern Enrique has used successfully across MaaS, comic_ai.rb and course_sketch.rb: a Ruby DSL that encapsulates orchestration complexity and reads like the domain (screenplay structure, shots, transitions), which compiles to a JSON manifest, which a Python engine executes against external AI services (LLM reasoning, image conditioning, HappyHorse video synthesis, and final assembly with FFmpeg). A Rails 8 application is the operator-facing surface: it accepts the prompt, runs the full pipeline as background jobs, and exposes progress/results.
The distinguishing technical bet of this submission is the 
hidden narrative engine: internally, script generation is not a blank-page LLM call. It is powered by a repurposed Myth Compiler — a DNA-extraction / semantic-projection pipeline that takes a base story's deep structure (archetypes, narrative genes, emotional curve) and transmutes it into a new domain. The hackathon judge never sees this: from their point of view, they type a prompt and get a drama. Internally, the prompt is classified against a curated catalog of base stories and destination domains, one pairing is selected (seeded, not purely random, so it is reproducible and explainable in the architecture diagram), and the Myth Compiler pipeline drives scriptwriting and storyboarding before HappyHorse ever sees a prompt.
This document specifies the full system: architecture, the Ruby DSL and its compiled manifest schema, the seven-stage pipeline, the Rails data model, the DashScope/HappyHorse integration contract, the token/credit budget strategy required by the track's judging criteria, and a six-week delivery plan.

3. System Architecture
3.1 Design Principles
	•	DSL-first: the orchestration surface is a Ruby DSL (ShowrunnerDSL), mirroring the comic_ai.rb / course_sketch.rb / video_sketch.rb pattern already proven in this codebase.
	•	Ontological separation: story DNA / catalog data, generated screenplay artifacts, and rendered media live in structurally separate tables and storage prefixes — never comingled, so a failed generation never corrupts the base catalog.
	•	Everything through Qwen: no OpenAI/Anthropic/Gemini calls in the critical path. A single QwenRouter adapter is the only egress point for LLM calls.
	•	Async-first: HappyHorse video synthesis is asynchronous (X-DashScope-Async: enable); the whole pipeline is modeled as a Sidekiq job chain with polling, not a blocking HTTP request.
	•	Token-budget-aware at every LLM call site — see Section 10.

3.3 Deployment Topology (Alibaba Cloud)
	•	Rails app + Sidekiq + Redis + Postgres: containerized, deployed on Alibaba Cloud ECS (or Alibaba Cloud Container Service for Kubernetes for the judged demo).
	•	Generated media (frames, video shots, final cuts): Alibaba Cloud OSS bucket, signed URLs for playback.
	•	All model calls (script, storyboard prompts, HappyHorse) go to dashscope-intl.aliyuncs.com — this alone satisfies "backend running on Alibaba Cloud" for the proof-of-deployment requirement, but the ECS/OSS hosting closes the loop completely.

4. The Hidden Narrative Engine (Myth Compiler Internals)
This is the section that must never surface in the product UI, demo copy, or public repo README beyond an internal code comment. From the judge's and end user's perspective there is one input — a prompt — and one output — a finished drama. Internally, the StoryEngine service performs the following, silently:
4.1 Why hide it
Two reasons drive this. First, product framing: the value proposition is "describe your drama, get your drama," not "pick a myth-transmutation domain." Second, and more importantly for the Innovation criterion: judges reward systems that feel like they reasoned about narrative structure, not systems that expose a menu of presets. The transmutation mechanism is the innovation; the prompt-to-selection mapping is the product.
4.2 Internal Pipeline (adapted from myth_compiler.rb)
	•	Prompt Intake — user's free-text prompt is embedded and classified (Qwen, low-token classification call) into: genre tone, protagonist archetype hint, conflict type, target emotional arc (tragic / hopeful / dark / epic).
	•	Base Story Selection — StoryCatalog holds a curated set of public-domain / archetype-level base stories (not verbatim copyrighted text — only structural DNA: protagonist/antagonist archetypes, narrative genes such as forbidden_love, betrayal, sacrifice, and an emotional curve). A weighted, seeded selection (seed = hash of the prompt) picks one base story so the same prompt is reproducible.
	•	Domain Selection — DomainCatalog holds destination domains (space_opera, cyberpunk_megacity, wild_west, feudal_japan, etc., taken from the existing domain list). Domain choice is conditioned on the classified tone/genre from step 1, then seeded-random among the compatible subset — this is the "random depending on the initial prompt" behavior the user asked for, but constrained so it never feels arbitrary.
	•	DNAExtractor / SemanticProjector / ArchetypeCompiler / WorldBuilder — reused conceptually from myth_compiler.rb's pipeline: project the base story's deep structure onto the chosen domain, producing a TransformMap and a CharacterRoster.
	•	NarrativeRecompiler — produces the actual scene-by-scene screenplay in the new domain, chapter/scene count driven by the short-drama duration target (e.g., 3 scenes × ~5–8s HappyHorse shots = a 60–90s drama).
	•	Handoff to Screenwriter/Storyboarder (Section 6) — StoryEngine's output is a StoryGraph + CompiledNarrative object; nothing about "myth compiling" appears past this boundary in any user-facing artifact.

5. ShowrunnerDSL — Ruby Orchestration Layer
Follows the exact shape of course_sketch.rb / video_sketch.rb: a Builder object collects declarative calls inside an instance_eval'd block and emits a JSON manifest. The DSL is intentionally thin — it does not itself talk to Qwen; it only describes the drama to be produced and the constraints (duration, resolution, token budget ceiling).
5.1 Public Entry Point
result = Showrunner.produce(
  prompt:        user_prompt,           # the ONLY creative input from the user
  output:        "drama_#{uuid}.mp4",
  target_duration: 75,                   # seconds, drives scene/shot count
  resolution:    "720P",                # HappyHorse: 480P | 720P | 1080P
  token_budget:  18_000,                 # hard ceiling across all Qwen calls
  video_model:   :happyhorse_1_1,         # maps to happyhorse-1.1-t2v / -i2v
  quality:       "high"
) do
  # nothing else required — everything below is auto-populated by StoryEngine,
  # but can be overridden for testing/determinism:
  seed 482913          # optional: pin story/domain selection for reproducible demos
end

6. Pipeline Stages (Detailed)
Stage 1 — Prompt Intake & Classification
Rails receives the prompt via ShowrunnerController#create, persists a Project record (status: queued), enqueues ShowrunnerPipelineJob. First engine step: one short QwenRouter call (qwen-turbo, ≤300 tokens) classifies tone/genre/conflict — cheapest model, tightest budget, per Section 10.
Stage 2 — Story & Domain Selection (hidden)
StoryEngine (Section 4) selects base_story_id and domain deterministically from the seed. Zero additional Qwen tokens spent here — this is pure Ruby/Python logic over the local catalog, which is precisely why hiding the transmutation strategy is cheap: it costs no tokens and adds narrative depth for free.
Stage 3 — Scriptwriting
Screenwriter calls qwen-plus with the TransformMap + CharacterRoster + target scene count, producing structured JSON (scene headings, action lines, dialogue) — response_format constrained to JSON, single call per scene batch (not per scene) to conserve tokens.
Stage 4 — Storyboarding
For each shot, Storyboarder derives a concise visual_prompt (reusing the scene's action/dialogue, compressed). For shots using i2v mode, a first-frame image is generated (Qwen image capability or a lightweight image model) and uploaded to OSS; for t2v-mode shots, no first frame is needed — HappyHorse's happyhorse-1.1-t2v generates directly from the prompt.
Stage 5 — Video Generation (HappyHorse)
VideoSynth submits one DashScope video-synthesis job per shot with X-DashScope-Async: enable, storing the returned task_id in video_jobs. A polling worker (Sidekiq, exponential backoff, 5s→30s) checks task status and downloads the result video to OSS on SUCCEEDED, retries with a simplified prompt on FAILED (max 2 retries per shot before flagging for manual review).
Stage 6 — Editing & Assembly
Editor stitches shots in scene order using an FFmpeg-driven Python renderer architecturally identical to video_sketch_renderer.py: transitions between shots (panel_slam, dissolve, wipe), optional caption overlays per scene heading, optional music/voice ducking exactly as VideoSketch's audio_track duck_when_voice model already implements.
Stage 7 — Delivery
Final MP4 uploaded to OSS, Project marked completed, Rails notifies the UI (ActionCable) with a signed playback URL and a budget_ledger summary shown to the user as "tokens & credits used" — itself a demo-worthy transparency feature for the Presentation criterion.

9.2 Polling
GET the task result endpoint using the returned task_id every 5s (backoff to 30s after 60s elapsed); statuses PENDING → RUNNING → SUCCEEDED/FAILED. On SUCCEEDED, stream the result URL into OSS via a background download job rather than holding the request open.
9.3 Failure Handling
	•	FAILED with content-policy reason → rewrite visual_prompt via qwen-turbo (strip flagged terms), retry once.
	•	FAILED with generic error → retry with unchanged prompt once, then mark shot needs_review and substitute a still-frame fallback so the pipeline still produces a complete cut.
	•	Timeout > 6 min per shot → abandon, use fallback still, log to token_ledger_entries with cost_estimate = 0 for transparency.
10. Token & Credit Budget Strategy
The track explicitly judges quality under a limited token budget. This is treated as a first-class constraint, not an afterthought:
	•	Every QwenRouter.call estimates tokens before sending (prompt length heuristic) and raises before exceeding the project's token_budget.
	•	Batch, don't loop: scriptwriting requests all scenes in one qwen-plus call with a strict JSON schema, rather than one call per scene — cutting call count (and system-prompt repetition cost) by the scene count.
	•	Cheapest-adequate model per stage (Section 8 table) — qwen-turbo for anything mechanical, qwen-plus only where creative quality matters, qwen-max reserved for rare fallback.
	•	Visual prompts are compressed (target ≤ 40 tokens) before being sent to HappyHorse — video generation cost/latency scales with prompt complexity far less than with duration/resolution, so the token savings here mainly protect the classification/scripting budget, not the video budget.
	•	budget_ledger is surfaced in the UI in real time so the judged demo can visibly show "used 6,400 / 18,000 tokens" — turning the constraint into a visible feature.
